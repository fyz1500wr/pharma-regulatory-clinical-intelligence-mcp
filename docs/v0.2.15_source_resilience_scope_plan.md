# v0.2.15 Source Resilience Scope Plan

## Purpose

v0.2.15 is a small source-resilience planning increment after `v0.2.14-fda-blocked-source-interpretation-regression`.

The goal is to align terminology and operator interpretation for degraded source-health states before making any additional runtime or test changes. This plan does not approve new sources, new MCP tools, live scraping workarounds, scheduler behavior, persistence, dashboards, alerts, or transport changes.

## Current Baseline

v0.2.14 confirmed that FDA `BLOCKED_SOURCE` / `SOURCE_UNAVAILABLE` must be interpreted as a source-health limitation, not as zero FDA regulatory results.

The baseline interpretation is:

- FDA source access may be blocked or unavailable in a live runtime environment.
- A blocked or unavailable FDA source must remain visible as a source-health issue.
- FDA source-health failure must not be silently converted into `FDA: 0 matching update(s)`.
- TFDA zero-result behavior remains distinguishable from FDA blocked-source behavior.
- Digest output remains working intelligence that requires human review before regulatory, clinical, legal, medical, competitive, commercial, or operational decisions.

## MVP Source Scope

MVP source scope remains limited to:

```text
FDA
TFDA
ClinicalTrials.gov
```

No v0.2.15 planning item should add or activate a new agency, registry, source family, connector path, or MCP tool.

## Recommended v0.2.15 Scope

The recommended v0.2.15 scope is deliberately narrow:

1. Source-health terminology alignment
   - Review existing documentation and tests for consistent use of `BLOCKED_SOURCE`, `SOURCE_UNAVAILABLE`, and `NO_MATCHING_RECORDS`.
   - Ensure wording does not imply that a blocked source is equivalent to a successful empty result.
2. Distinction between source-health and query outcomes
   - Clarify that `BLOCKED_SOURCE` means access was blocked by the source or runtime egress environment.
   - Clarify that `SOURCE_UNAVAILABLE` means the source could not be reached or used successfully.
   - Clarify that `NO_MATCHING_RECORDS` means the source/query path succeeded but no records matched the requested criteria.
3. Operator guidance for degraded source-health states
   - Document how operators should interpret partial results when FDA, TFDA, or ClinicalTrials.gov is degraded.
   - Reinforce that source-health warnings must be reviewed before relying on digest or comparison output.
   - Keep guidance informational; do not introduce final regulatory, clinical, legal, medical, commercial, or business conclusions.
4. Deterministic offline regression only if an ambiguity is found
   - Add or adjust a small offline regression only if the terminology review identifies an actual ambiguity not already covered by tests.
   - Keep any regression deterministic and offline.
   - Do not depend on live FDA, TFDA, or ClinicalTrials.gov availability.

## Out of Scope

The following are explicitly out of scope for v0.2.15 unless a later approved scope document changes the plan:

- New sources or source families.
- New MCP tools or renamed MCP tools.
- EMA, NMPA, PMDA, WHO ICTRP, or EU CTIS.
- Literature, patent, or finance integrations.
- Scheduler, alerts, persistence, dashboard, HTTP/SSE transport, or GitHub automation.
- `.mcp.json` changes.
- Live scraping workarounds.
- Bypassing source controls, runtime egress controls, access controls, rate limits, or terms of use.
- Runtime behavior changes that alter connector behavior, MCP tool contracts, or normalized record semantics.

## Preferred Implementation Path

Use this order for the next v0.2.15 work item:

```text
documentation review
  → terminology gap identification
  → only then consider a small offline regression
  → project-state update
  → focused tests
  → full tests
```

Implementation guidance:

- Start by reviewing README, source-health documentation, project state, and relevant tests.
- Identify exact wording gaps before editing.
- Prefer documentation-only changes when wording alignment is sufficient.
- Add a deterministic offline regression only when an ambiguity could cause a blocked or unavailable source to be interpreted as zero matching records.
- Keep all changes inside MVP v1 source scope.

## Decision Criteria

Proceed with a v0.2.15 change only if at least one of these criteria is met:

- Existing documentation uses inconsistent terminology for `BLOCKED_SOURCE`, `SOURCE_UNAVAILABLE`, or `NO_MATCHING_RECORDS`.
- Operator-facing guidance does not clearly distinguish source-health failure from successful no-result outcomes.
- A deterministic offline test gap could allow FDA `BLOCKED_SOURCE` / `SOURCE_UNAVAILABLE` to be interpreted as zero FDA regulatory results.
- Project-state language needs a controlled update to preserve the post-v0.2.14 interpretation.

## Stop Conditions

Stop or defer the work if the next change would require any of the following:

- Adding a new source, agency, registry, source connector, or source family.
- Adding or renaming an MCP tool.
- Changing `.mcp.json`.
- Adding scheduler, alerts, persistence, dashboard, HTTP/SSE transport, or GitHub automation.
- Adding EMA, NMPA, PMDA, WHO ICTRP, EU CTIS, literature, patent, or finance integrations.
- Implementing live scraping workarounds or bypassing access controls.
- Making runtime changes without a documented ambiguity and focused test plan.
